Anexo: Guía de Configuración y Despliegue de Infraestructura en AWS (RoomOps)

Manual técnico de referencia para el aprovisionamiento de servicios y puesta en marcha del entorno de producción utilizando el grupo de seguridad por defecto.

Este manual técnico descrita el proceso completo desde cero para la creación, configuración y despliegue de la arquitectura web en Amazon Web Services (AWS), utilizando una base de datos relacional (RDS PostgreSQL), dos instancias de servidores virtuales (EC2) para Frontend y Backend, y la adecuación de las reglas de entrada sobre el grupo de seguridad por defecto (default).

1. Creación y Configuración de la Base de Datos (AWS RDS)

1.1 Crear la instancia de base de datos

  1. Buscar el servicio RDS en la barra superior de la consola de AWS.
  2. Hacer clic en el botón Create database (Crear base de datos).
  3. Seleccionar el método de creación Standard create (Creación estándar).
  4. En las opciones de motor (Engine options), seleccionar PostgreSQL.
  5. En las plantillas (Templates), elegir Free Tier (Capa gratuita) para evitar costos asociados.
  6. En la sección de configuración (Settings):
    • Asignar un identificador a la base de datos (ej. roomops-db).
    • Definir el nombre de usuario maestro (por defecto suele ser postgres).
    • Establecer y guardar en un lugar seguro la contraseña maestra.
  7. En la sección de conectividad (Connectivity):
    • Configurar el acceso público (Public access) en Yes (Sí), para permitir las pruebas de conexión iniciales desde el entorno de desarrollo local.
  8. En la configuración adicional (Additional configuration):
    • Definir el nombre de la base de datos inicial (ej. roomops).
  9. Hacer clic en Create database. El proceso de aprovisionamiento puede tardar unos minutos hasta que el estado cambie a Available (Disponible).
  10. Copiar el Endpoint (Punto de enlace) generado, el cual servirá como dirección del host en la configuración del Backend.

2. Creación de Instancias Virtuales (AWS EC2)

Se requiere la creación de dos instancias separadas con el sistema operativo Debian o Ubuntu.

2.1 Crear la Instancia para el Backend

  1. Buscar el servicio EC2 en la consola de AWS.
  2. Hacer clic en Launch instance (Lanzar instancia).
  3. Asignar el nombre RoomOps-Backend en la sección de etiquetas.
  4. En las imágenes de sistema operativo (Application and OS Images), seleccionar Debian (o Ubuntu).
  5. En el tipo de instancia (Instance type), seleccionar t2.micro (o t3.micro, según disponibilidad de la capa gratuita).
  6. En el par de claves (Key pair), seleccionar o crear una clave .pem para permitir la posterior conexión SSH mediante Remmina.
  7. En la configuración de red (Network settings), seleccionar el grupo de seguridad por defecto (default).
  8. Hacer clic en Launch instance.

2.2 Crear la Instancia para el Frontend

  1. Repetir el proceso anterior haciendo clic en Launch instance.
  2. Asignar el nombre RoomOps-Frontend.
  3. Seleccionar la misma distribución de sistema operativo y tipo de instancia (t2.micro).
  4. Seleccionar el mismo par de claves .pem.
  5. En la configuración de red, seleccionar igualmente el grupo de seguridad por defecto (default).
  6. Hacer clic en Launch instance.

3. Configuración de Red y Reglas del Grupo de Seguridad por Defecto

Es fundamental abrir los puertos correctos en el grupo de seguridad compartido por los recursos para permitir el flujo de datos entre el cliente externo, el Frontend, el Backend y la Base de Datos.

3.1 Configurar Reglas de Entrada (Inbound Rules) en el Grupo Default

  1. Ir a la sección de Security Groups dentro del menú izquierdo de EC2.
  2. Seleccionar el grupo de seguridad llamado default asignado a la base de datos RDS y a ambas instancias EC2.
  3. En la pestaña Inbound rules (Reglas de entrada), hacer clic en Edit inbound rules.
  4. Asegurar y añadir las siguientes reglas de entrada para habilitar la conectividad requerida:
    • Regla 1 (SSH): Puerto 22 | Source: 0.0.0.0/0 o la IP del administrador (Permite el acceso a terminales SSH y SFTP desde Remmina).
    • Regla 2 (PostgreSQL): Puerto 5432 | Source: 0.0.0.0/0 (Habilita el acceso externo a la base de datos RDS).
    • Regla 3 (Custom TCP): Puerto 5000 | Source: 0.0.0.0/0 (Permite al navegador consumir la API REST del Backend y visualizar la interfaz de Swagger).
    • Regla 4 (HTTP): Puerto 80 | Source: 0.0.0.0/0 (Habilita el acceso público de usuarios al servidor HTTP Nginx del Frontend).
  5. Hacer clic en Save rules (Guardar reglas) para aplicar los cambios de red en caliente a todos los servicios asociados.

4. Despliegue y Ejecución del Backend (Spring Boot)

4.1 Preparación del código local

Antes de compilar, asegurar que el archivo application.properties o application.yml contenga el Endpoint de la RDS, las credenciales maestras configuradas en el punto 1, y que el archivo SecurityConfig.java incluya la IP pública del Frontend en los orígenes permitidos de CORS.

Ejecutar la compilación local en la terminal del Backend:

mvn clean package

4.2 Despliegue en el servidor EC2

Conectarse por SSH a la instancia de Backend mediante Remmina y transferir el archivo .jar generado (ubicado en la carpeta target/) a través del panel SFTP.

Ejecutar los siguientes comandos en la terminal de la instancia EC2 para limpiar procesos previos y levantar el servicio en segundo plano:

# Liberar de forma preventiva el puerto 5000 ante cualquier ejecución colgada sudo kill -9 $(sudo lsof -t -i:5000) 2>/dev/null || true # Iniciar la aplicación Spring Boot en segundo plano utilizando nohup nohup java -jar nombre-del-archivo-backend.jar > backend.log 2>&1 & # Verificar la correcta asignación del puerto de escucha sudo ss -tlnp | grep :5000
Nota: Ejecutar opcionalmente tail -f backend.log para visualizar los registros de inicialización de la API en tiempo real. Se puede salir de dicha vista utilizando la combinación Ctrl + C sin detener el servicio de fondo.

5. Despliegue y Ejecución del Frontend (Nginx + React/Vite)

5.1 Preparación del código local

Asegurar que el archivo de configuración del cliente (ej. api.js) apunte a la dirección IP pública y puerto 5000 de la instancia del Backend.

Ejecutar la compilación local en la terminal del Frontend:

npm run build

(Este comando genera una carpeta llamada dist en la raíz del proyecto).

5.2 Configuración del Servidor Web Nginx en EC2

Conectarse por SSH a la instancia de Frontend mediante Remmina y ejecutar la instalación del servidor:

# Actualizar el índice de paquetes del sistema operativo sudo apt update # Instalar el servidor web Nginx sudo apt install nginx -y # Habilitar e iniciar el servicio web sudo systemctl enable nginx sudo systemctl start nginx # Otorgar privilegios al usuario actual sobre la carpeta raíz de despliegue sudo chown -R $USER:$USER /var/www/html # Eliminar el archivo por defecto de bienvenida de Nginx rm -f /var/www/html/index.nginx-debian.html

5.3 Transferencia de archivos estáticos

Mediante el panel SFTP de Remmina, entrar a la carpeta local dist, seleccionar todo su contenido interno (los archivos sueltos y la carpeta assets) y arrastrarlos directamente hacia el directorio remoto /var/www/html/.

5.4 Evitar el error 404 ante recargas de rutas virtuales

Editar el archivo de configuración de Nginx para redirigir el tráfico dinámico al punto de entrada de la SPA:

sudo nano /etc/nginx/sites-available/default

Localizar el bloque de directivas location / y sustituir el valor =404 por /index.html. Modificar la estructura para que se lea exactamente de la siguiente forma:

location / { try_files $uri $uri/ /index.html; }

(Guardar la edición mediante la combinación de teclas Ctrl + O, confirmar con Enter y salir del editor con Ctrl + X).

5.5 Reinicio y confirmación del Frontend

Validar la configuración y aplicar los cambios estructurales en el servidor web:

# Validar la sintaxis del archivo modificado sudo nginx -t # Reiniciar el servicio web para aplicar las nuevas rutas sudo systemctl restart nginx # Comprobar el estado activo y saludable del servicio sudo systemctl status nginx

A partir de este punto, la arquitectura se encuentra totalmente desplegada y en funcionamiento en entorno de producción.
